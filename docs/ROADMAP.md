# Дорожная карта проекта «Генератор ТКП»

Документ для планировщика (sub-agent): разбивка проекта на разделы и подразделы с целью постановки задач разработке и контроля кода.

---

## 1. ТКП (Технико-коммерческие предложения)

### 1.1 Простое ТКП (одна услуга)
- Форма создания: дата, услуга, регион, заказчик, помещение, площадь, сроки.
- Подтверждение и генерация DOCX/PDF из шаблонов `templates_docx/`.
- Сохранение записи `TKPRecord`; привязка к контрагенту (опционально).
- Скачивание файла и страница успеха (`download_success`); отправка PDF на email.

### 1.2 Комплексное ТКП
- Форма с несколькими услугами и таблицей (`complex_form`, `complex_confirm`).
- Хранение состава в `TKPRecord.rows_json`.
- Шаблон «Шаблон 9 Комплексное ТКП.docx»; подстановка таблицы спецификации.
- Комментарии к услугам: `data/complex_service_comments.json`.

### 1.3 Список и отображение ТКП
- Таблица ТКП (`table_view`, `table.html`): фильтры, сортировка, скачивание по типу (DOCX/PDF).
- Модальное окно отправки PDF на email (`send_tkp_pdf_email_view`).
- Валидация имени файла/базового имени ТКП (`_validate_tkp_base_name`).

### 1.4 Услуги и тарифы
- Модели: `Service`, `Region`, `RegionServicePrice`.
- Команды: `init_services` (создание/обновление услуг из шаблонов), `load_region_prices` (CSV → тарифы).
- Страницы: тарифы (`tariffs`), описания услуг (`service_descriptions`).
- Данные: `region_price.csv`, шаблоны в `templates_docx/`.

### 1.5 Шаблоны DOCX для ТКП
- Каталог `templates_docx/`: шаблоны 1–9 и др.; соответствие услугам в коде/команде `init_services`.
- Логика выбора шаблона при генерации; плейсхолдеры docxtpl.

## 2. Договоры

### 2.1 Создание договора по ТКП
- Форма договора (`contract_form`): выбор шаблона по услуге/комплексному ТКП (`SERVICE_TO_CONTRACT_TEMPLATE`).
- Реквизиты заказчика (из контрагента или ручной ввод); выпадающие списки: `customer_in_person`, `acting_on_basis`, `work_completion_period`, `period_starts_from`.
- Генерация DOCX (docxtpl) → конвертация в PDF; сохранение `ContractRecord` и снимка полей (`contract_snapshot`).
- Кнопка и модальное окно отправки договора на email (если реализовано).

### 2.2 Шаблоны договоров и плейсхолдеры
- Каталог `templates_docx/contracts_templates/`: 01–08 (ДП, ДК, Навигация, Контент, комплексные, благоустройство).
- Спецплейсхолдер `___CONTRACT_SPEC_TABLE___` для таблицы спецификации в комплексных договорах.
- Справочник полей: `contract_fields.py`, `data/contract_placeholders.txt`.

### 2.3 Редактор договора (без привязки к ТКП)
- Создание/редактирование договора через `contract_editor_view`, `contract_save_from_editor_view`.
- Сохранение/обновление `ContractRecord` и снимка.

### 2.4 Список и карточка договора
- Таблица договоров (`contract_table_view`); карточка договора (`contract_card_view`, `contract_card_content.html`).
- Скачивание DOCX/PDF (`contract_download_file_view`), страница успеха (общая с ТКП при необходимости).

---

## 3. Контрагенты и реквизиты

### 3.1 Модель и хранение
- Модель `Counterparty`: реквизиты (наименование, ИНН, КПП, адрес, банк, контакты и т.д.).
- Связь с договорами и снимками; опциональная привязка ТКП к контрагенту (если используется).

### 3.2 Добавление реквизитов
- Страница добавления реквизитов (`requisites_add_view`, `requisites_form.html`): загрузка DOCX/PDF.
- Парсер `requisites_parser.py`: извлечение реквизитов из документа → сохранение `Counterparty`.
- Форма `RequisitesParseForm`; использование в контрактах (подстановка в форму договора).

### 3.3 Справочник контрагентов
- Список контрагентов (`counterparties_view`, `counterparties.html`).
- Поиск (`counterparty_search_view`); JSON по одному контрагенту (`counterparty_json_view`) для подстановки в формы.
- Навигация: доступ к «Добавление реквизитов» из раздела контрагентов/договоров (не из главного меню).

## 4. Канбан-доска

### 4.1 Модели и порядок
- Модели: `KanbanColumnTitleOverride`, `KanbanColumnCustom`, `KanbanCardPosition`, `KanbanCardField`, `KanbanBoardOrder`.
- Колонки: черновик, финал, черновик договора, финал договора, пользовательские.

### 4.2 Представление и интерфейс
- Страница канбана (`kanban_view`, `kanban.html`); контент карточки (`kanban_card_content.html`).
- Перетаскивание карточек (AJAX/JSON); изменение заголовков колонок; пользовательские колонки и поля.

### 4.3 API-подобные эндпоинты (JSON)
- Обновление заголовка колонки; переупорядочивание колонок; перемещение карточки; заметки по карточке; сохранение поля карточки.
- Все под `@login_required`; возврат JSON для обновления UI без перезагрузки.

## 5. Аутентификация и доступ

### 5.1 Вход и выход
- Django auth: `accounts/login`, `accounts/logout`; `LOGIN_URL` в настройках.
- Шаблон `registration/login.html`; редирект после входа.

### 5.2 Ограничение доступа
- Все представления приложения `proposals` под `@login_required`.
- Учёт создателя записей: поля `created_by` на TKP и Contract (миграции 0018), если используются для разграничения.

## 6. Администрирование (Django Admin)

### 6.1 Зарегистрированные модели
- Регион, тарифы: `Region`, `RegionServicePrice`, `Service`.
- ТКП и контрагенты: `TKPRecord`, `Counterparty`.
- Модели договоров и канбана в админку не вынесены (при необходимости — отдельная задача).

### 6.2 Настройки и права
- Настройки админки в `proposals/admin.py`; при необходимости — кастомизация списков, фильтров, полей только для чтения.

## 7. Инфраструктура и развёртывание

### 7.1 Проект Django
- `tkp_generator`: `settings.py`, `urls.py`, `wsgi.py`, `asgi.py`.
- Подключение приложения `proposals`; маршрутизация корня в `proposals.urls`.
- Настройки: SQLite, язык `ru-ru`, таймзона Москва; каталоги `TEMPLATES_DOCX_DIR`, `TKP_OUTPUT_DIR`; SMTP (Yandex).

### 7.2 Окружение и секреты
- Переменные: `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS` (чтение из `os.environ`).
- Файл `.env.example` для документации; при необходимости — подключение python-dotenv.

### 7.3 Развёртывание на сервере
- Скрипт `deploy.sh`: venv, установка зависимостей, миграции, collectstatic, перезапуск Gunicorn.
- Документация: `DEPLOY.md`, `DEPLOY_VM.md` (Yandex Cloud VM, Ubuntu, Nginx, Gunicorn, LibreOffice).
- Конвертация PDF: LibreOffice (Linux) или docx2pdf + MS Word (Windows).

### 7.4 Статика и медиа
- `static/`, `images/`: логотипы, favicon, изображения; `STATICFILES_DIRS`; в проде — `collectstatic` в `staticfiles/`.

## 8. Шаблоны (HTML) и фронтенд

### 8.1 Базовый шаблон и навигация
- `proposals/templates/proposals/base.html`: общий каркас, верхняя панель (мобильная), боковое меню (десктоп).
- Пункты меню: Старт, Инструкция, Перечень ТКП, Канбан, Договоры, Контрагенты, Тарифы, Описания услуг, Инструкция (ссылка). Пункт «Добавление реквизитов» убран из меню; доступ через Контрагенты/Договоры.

### 8.2 Шаблоны страниц
- ТКП: `start.html`, `instruction.html`, `form.html`, `confirm.html`, `download_success.html`, `complex_form.html`, `complex_confirm.html`, `table.html`.
- Договоры: `contract_table.html`, `contract_form.html`, `contract_editor.html`, `contract_card_content.html`, `requisites_form.html`.
- Справочники: `tariffs.html`, `service_descriptions.html`, `counterparties.html`.
- Канбан: `kanban.html`, `kanban_card_content.html`.
- Регистрация: `registration/login.html`.

### 8.3 Шаблонные теги и фильтры
- `templatetags/proposal_filters.py`: например `get_item`, `format_price`; использование в шаблонах.

### 8.4 Клиентская логика
- Минимальный JS: канбан (перетаскивание, запросы), форма договора/редактор, поиск контрагентов, модальные окна (email и т.д.).

## 9. Данные и конфигурация

### 9.1 Исходные данные
- `data/`: `contract_placeholders.txt`, `complex_service_comments.json`, прочие JSON/текстовые справочники.
- `region_price.csv` — загрузка тарифов.
- `clients_cards/` — примеры DOCX/PDF для тестов парсера реквизитов.

### 9.2 Управляющие команды
- `init_services`: создание/обновление услуг из шаблонов ТКП.
- `load_region_prices`: импорт тарифов из CSV в `RegionServicePrice`.
- При необходимости: экспорт/импорт, бэкапы, очистка старых файлов в `TKP_output/`.

## 10. Качество кода и технический долг

### 10.1 Тестирование
- Наличие/отсутствие unit- и интеграционных тестов; покрытие представлений, форм, парсера реквизитов, генерации документов.

### 10.2 Рефакторинг и единообразие
- Разделение крупных `views.py` по доменам (ТКП, договоры, канбан, контрагенты) или вынос части логики в сервисы/хелперы.
- Единый стиль обработки ошибок и сообщений пользователю; логирование критичных операций.

### 10.3 Безопасность и валидация
- Валидация загружаемых файлов (тип, размер); санитизация данных для docxtpl и вывода в HTML.
- Проверка прав при скачивании/отправке файлов (доступ только к своим или по ролям, если появятся роли).

## Как пользоваться этой картой

- **Планировщик** может назначать задачи по разделам (например: «2. Договоры → 2.1», «4. Канбан → 4.2»).
- В каждом подразделе перечислены компоненты (views, модели, файлы); по ним можно формулировать конкретные тикеты: доработка, багфикс, рефакторинг, тесты.
- Новые крупные фичи стоит сначала отразить в соответствующем разделе/подразделе, затем дробить на задачи для разработки и контроля.
